Estrategias para el diseño de pruebas

Estrategia de pruebas

Una estrategia de prueba de software proporciona una guía que describe los pasos que deben realizarse como parte de la prueba, cuándo se planean y se llevan a cabo dichos pasos, y cuánto esfuerzo, tiempo y recursos se requerirán.
Una estrategia de prueba de software debe ser suficientemente flexible para promover un uso personalizado de la prueba. Al mismo tiempo, debe ser suficientemente rígida para alentar la planificación razonable y el seguimiento de la gestión conforme avanza el proyecto.
Para poder llevar a cabo un control de calidad, es imprescindible haber definido los requisitos del sistema de software que vamos a implementar, ya que son necesarios para disponer de una especificación del propósito del software. Los procesos de calidad verifican que el software cumple con los requisitos que definimos originalmente.
Características Generales:

  1. Deben comenzar desde cada componente del sistema (desde el código, por ejemplo), y deben de llegar hasta la integración total del mismo, es decir, comprobar que se cumple con los requerimientos iniciales del sistema.
  2. Las técnicas variaran en función al enfoque y tiempo requerido para cada prueba
  3. Esta es una actividad que debe de ser realizada por el Desarrollador del software, o en todo caso, por un grupo de prueba independiente (si se trata de un sistema mayor).
  4. Se debe incluir las depuraciones
  5. Se deben de realizar las correspondientes revisiones técnicas para localizar y eliminar errores de tal manera que se pueda facilitar esta actividad


La prueba de software es un elemento de un tema más amplio que usualmente se conoce como verificación y validación (V&V). La verificación se refiere al conjunto de tareas que garantizan que el software implementa correctamente una función específica. La validación es un conjunto diferente de tareas que aseguran que el software que se construye sigue los requerimientos del cliente


Estrategias para el diseño de casos pruebas.


Las estrategias de prueba deben de garantizar que tras haber realizado dicha actividad se habrán encontrado y corregido todos los errores que contenga nuestro sistema, sin haber gastado tantos recursos o tiempo, como ya se mencionó antes.

Enfoques principales.


  1. Estructural: también lo conocemos como pruebas de caja blanca, por ser una actividad exhaustiva en donde probamos nuestro sistema a nivel de código (Componente a componente), revisando todas las posibles entradas y salidas para cada componente.
  2. Funcional: Es llamado prueba de caja negra, ya que solamente se comprueba la funcionabilidad total del sistema y no por componentes, introduciendo todas las posibles entradas aceptadas por él y verificando las salidas arrojadas.
  3. Aleatorio: Para este enfoque utilizaremos modelos estadísticos que nos permitan seleccionar las posibles entradas para nuestro sistema y del mismo modo podamos predecir las posibles salidas que arrojara.
